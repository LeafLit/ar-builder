# 自定义状态名称设计

## 背景

AR Builder 当前已经支持采集两个状态、训练识别模型、绑定文字/图片/3D 模型/音效，并在测试页按识别结果触发输出。现在的问题是界面仍然大量显示固定文案“状态 A / 状态 B”。这对小白用户不直观：他们现场实际训练的是“拳头 / 巴掌”“杯子在左边 / 杯子在右边”，但后续训练、编辑、测试页仍要靠 A/B 记忆含义。

PRD 里也提到用户需要创建“命名状态”。本轮先实现最小闭环：保持两个固定状态和内部 ID 不变，只允许用户修改显示名称。

## 目标

1. 用户可以在采集页给两个状态输入名称，例如“拳头”和“巴掌”。
2. 自定义名称贯穿采集、训练、编辑、测试、状态计数和缺少绑定提示。
3. 保存本地项目时保留状态名称；重新打开项目后继续显示自定义名称。
4. 旧项目没有自定义名称时继续使用默认“状态 A / 状态 B”。
5. 内部状态 ID 仍然是 `state_a` 和 `state_b`，避免影响样本、模型、绑定、素材和已保存项目。

## 非目标

1. 本轮不做任意新增/删除状态。
2. 本轮不修改训练算法、识别阈值、连续多帧确认机制。
3. 本轮不迁移素材 ID 或绑定 ID。
4. 本轮不做登录、云同步或多人协作。

## 方案

### 数据结构

在应用状态中增加 `states` 字段，保存两个可编辑状态的元数据：

```ts
type AppInputState = {
  id: string;
  name: string;
  order: number;
};
```

默认值为：

```ts
[
  { id: "state_a", name: "状态 A", order: 0 },
  { id: "state_b", name: "状态 B", order: 1 }
]
```

新增一个状态规范化工具，集中处理默认状态、旧项目兜底、名称清理和按顺序输出。它只保留当前 MVP 支持的两个固定状态 ID；如果旧数据缺少某个状态，就补回默认状态。

### 状态重命名

新增 reducer action：

```ts
{ type: "renameState"; stateId: string; name: string }
```

规则：

1. 输入会 `trim()`。
2. 非空时更新对应状态名称。
3. 空名称不覆盖已有名称，避免用户误删后整个流程出现空标签。
4. 未知 `stateId` 不改变应用状态。

### 保存与恢复

`createProjectFromAppState` 使用 `state.states` 生成 `Project.states`，并继续根据 `sampleCounts` 生成 `sampleIds`。

`restoreStateFromProject` 和 `loadProject` 读取 `project.states` 里的名称，再通过规范化工具补齐默认状态。旧项目没有设置或缺少字段时仍恢复为“状态 A / 状态 B”。

### 页面串接

`App` 将 `state.states` 传给四个关键页面：

1. 采集页：显示名称输入框、状态按钮、采集成功提示。
2. 训练页：样本数量汇总、缺少样本提示使用自定义名称。
3. 编辑页：输出类型、文字、图片、3D 模型、音效、锚点控件的标签使用自定义名称。
4. 测试页：模拟识别按钮、当前识别状态、状态计数、缺少绑定提示使用自定义名称。

各页面保留默认状态兜底，保证单独测试或旧调用方式不崩。

## 用户体验

采集页在“选择状态”区域增加两个简短输入框。默认值是“状态 A / 状态 B”。用户可以直接改成“拳头 / 巴掌”，再采集样本。

后续所有页面都显示这些名称。例如：

1. 训练页显示“拳头：3 个样本”。
2. 编辑页显示“拳头 的 AR 文字”。
3. 测试页按钮显示“模拟识别拳头”。
4. 计数面板显示“拳头 触发 1 次”。

## 兼容性

1. 现有项目保存结构已经有 `states`，本轮只是让应用状态真实使用其中的 `name`。
2. 已保存项目中的 `sampleIds`、`assets`、`bindings`、`recognitionModel` 不需要迁移。
3. 默认状态 ID 不变，因此历史绑定和测试识别结果仍然能匹配。
4. 旧项目缺失名称时使用默认名称。

## 测试策略

1. `appState`：验证初始状态包含默认名称、重命名成功、空名称不覆盖、加载项目恢复名称。
2. `appProjectSnapshot`：验证保存项目会写入自定义名称，恢复项目会带回名称。
3. `CaptureScreen`：验证输入名称会触发回调，状态按钮和采集提示能显示新名称。
4. `TrainScreen`：验证样本汇总和缺少样本提示使用新名称。
5. `AuthoringScreen`：验证编辑控件标签使用新名称且保存输出仍按 `state_a/state_b`。
6. `TestScreen`：验证模拟识别、当前识别、缺少绑定提示和计数使用新名称。
7. `App`：验证从采集页改名后，后续页面继续显示改名结果。

## 风险与应对

1. 风险：状态名称变更后误影响内部 ID。
   应对：测试断言保存输出、绑定和识别仍使用 `state_a/state_b`。

2. 风险：旧项目打开后状态缺失。
   应对：规范化工具统一补齐默认状态。

3. 风险：界面输入名称为空造成后续标签空白。
   应对：空名称不提交覆盖；界面仍保留当前名称。

4. 风险：多个页面本地硬编码默认状态导致显示不一致。
   应对：抽出共享状态工具，各页面通过 props 接收状态并保留兜底。
