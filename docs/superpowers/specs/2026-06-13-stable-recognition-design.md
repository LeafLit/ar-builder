# 连续多帧确认机制设计

## 背景

AR Builder 当前实时测试页已经支持真实摄像头自动识别、识别灵敏度滑杆和项目级保存灵敏度。现在的显示逻辑是：某一帧预测结果达到灵敏度阈值后，立即显示该状态绑定的虚拟物体；低于阈值时不显示。这种单帧判定在真机现场容易受光线、手势边缘和相机画面变化影响，出现虚拟物体闪烁或误切换。

## 目标

- 自动识别结果连续稳定后再显示虚拟物体。
- 减少低置信度、单帧误判、状态来回跳导致的闪烁。
- 保持“未识别到已训练状态时不显示虚拟物体”的原则。
- 模拟识别按钮仍然即时显示，方便调试状态绑定和输出内容。
- 先用固定默认值，不增加新的 UI 设置项。

## 非目标

- 不调整 MobileNet embedding 或分类器算法。
- 不改变识别灵敏度滑杆的范围、默认值和保存逻辑。
- 不新增项目设置字段。
- 不改变摄像头采样间隔。
- 不处理多状态扩展之外的复杂投票策略。

## 推荐方案

采用 **2 帧确认 + 2 帧丢失清空**：

- 同一个状态连续 2 次达到当前灵敏度阈值后，才成为已确认状态。
- 如果预测状态切换，需要新状态重新连续达到 2 次，才替换当前显示。
- 如果预测结果低于阈值，计入“丢失帧”。
- 连续 2 次低于阈值后，清空已确认状态，不显示虚拟物体。
- 只作用于自动识别；模拟识别按钮直接设置确认状态。

这个策略在当前识别间隔约 350ms 的基础上，通常会在约 700ms 内确认显示。它比单帧稳定，又不会像 3 帧确认那样明显拖慢现场反馈。

## 设计

新增一个小模块：

```text
src/features/testing/stableRecognition.ts
```

模块提供纯函数和状态类型，负责把自动识别的一帧帧结果转成“已确认可显示的状态”。

核心概念：

```ts
type StableRecognitionInput = {
  stateId?: string;
  confidence?: number;
};

type StableRecognitionState = {
  candidateStateId?: string;
  candidateCount: number;
  confirmedStateId?: string;
  missedCount: number;
};
```

核心行为：

- `stateId` 存在且 `confidence` 达到阈值时，更新候选状态。
- 候选状态连续达到 `requiredMatches` 次后，写入 `confirmedStateId`。
- 不达阈值或没有状态时，增加 `missedCount`。
- `missedCount` 达到 `requiredMisses` 后，清空 `confirmedStateId` 和候选状态。
- 默认配置为 `{ requiredMatches: 2, requiredMisses: 2 }`。

`TestScreen` 继续接收原始识别预测，但不再把每一帧直接作为显示状态。自动识别回调会先更新稳定器状态，再根据 `confirmedStateId` 找到对应状态并显示输出。状态文案中的置信度仍显示最近一帧自动识别置信度，方便现场观察。

模拟识别按钮会跳过稳定器，直接显示对应状态，并停止自动识别会话；这与现在行为一致。

## 状态文案

现有文案基本保持：

- 还没确认状态但有识别帧时，继续显示“自动识别中，未识别到已训练状态。”
- 已确认状态后，显示“当前识别：状态 A（xx%）”。
- 启动中、失败、等待结果等文案不变。

后续如果小白用户需要更清楚的提示，可以再加一句“正在确认识别结果”，但本次不增加新文案，避免界面变复杂。

## 测试计划

- `stableRecognition` 单元测试：
  - 单帧达到阈值不会确认显示。
  - 同一状态连续 2 帧达到阈值后确认显示。
  - 切换状态时需要新状态连续 2 帧才切换显示。
  - 连续 2 帧低于阈值后清空确认状态。
  - 低置信度不会累积候选确认。
- `TestScreen` 组件测试：
  - 自动识别第一帧达到阈值时不显示虚拟物体。
  - 自动识别连续两帧同状态达到阈值后显示虚拟物体。
  - 连续低置信度后清空显示。
  - 模拟识别按钮仍然即时显示。

## 验收标准

- 拳头/巴掌自动识别时，虚拟物体不因单帧误判快速闪烁。
- 已识别状态丢失后，连续约 2 帧未达阈值会隐藏虚拟物体。
- 灵敏度滑杆仍然即时影响阈值。
- 项目保存和重新打开灵敏度的功能不回退。
- `pnpm.cmd test` 和 `pnpm.cmd run build` 通过。
